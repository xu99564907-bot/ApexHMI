# 开放式 HMI 软件平台 — 开发需求与架构说明

> **文档目的**：将「替代威纶通、西门子等厂商人机界面」的长期目标，落实为可迭代的产品需求与架构约束，指导后续版本规划与实现边界划分。  
> **当前工程基线**：`ApexHMI`（.NET 8、WPF、OPC UA、MVVM），已具备 IO 表导入、按规则生成 PLC 程序片段、并据此刷新气缸/轴等功能块 UI 的雏形。

---

## 1. 产品定位与愿景

### 1.1 一句话定位

构建一套 **开放、可配置、可扩展** 的工业人机交互软件平台：在 **不修改产品源码** 的前提下，终端用户或集成商能够 **增删页面与控件**、 **按业务规则批量生成界面与控件**，并与现场 PLC/OPC UA 等数据源可靠对接。

### 1.2 与传统专有 HMI 的差异


| 维度   | 典型专有 HMI        | 本平台目标                         |
| ---- | --------------- | ----------------------------- |
| 界面修改 | 厂商组态软件 + 专有工程格式 | 开放配置（如 JSON/数据库）+ 可选可视化组态     |
| 源码依赖 | 深度绑定厂商生态        | 运行时加载定义，与主程序解耦                |
| 规则生成 | 部分支持脚本/宏，封闭     | **显式规则引擎 + 模板**，可版本化、可测试      |
| 数据接入 | 驱动列表由厂商决定       | **开放式适配层**（当前以 OPC UA 为主，可扩展） |


### 1.3 成功标准（验收视角）

1. **零源码改 UI**：用户通过配置或组态工具完成页面与控件的增删改，发布后即可在客户端生效（在权限与许可证允许范围内）。
2. **页面生命周期可控**：支持新增、复制、删除、排序页面；删除页面不破坏其余页面与全局导航。
3. **规则驱动生成**：输入结构化数据（如 IO 表、设备清单、配方结构），按 **可配置的规则与模板** 生成界面树、控件实例及数据绑定，行为与现有「IO 导入 → 气缸/轴块」一致且可扩展到其他对象类型。
4. **可维护与可升级**：平台发新版时，用户工程可迁移；规则与模板可独立演进。

## 2. 范围与非目标

### 2.1 范围内

- 运行时 **UI 定义模型**（页面、布局、控件类型、属性、绑定、样式引用）。  
- **控件库契约**：标准控件与行业控件（按钮、指示灯、数值、趋势、报警视图、气缸/轴等功能块）的统一元数据与行为接口。  
- **规则引擎与模板**：从外部数据生成 UI/绑定/脚本的流水线（与现有 `IoProgramGenerationService`、IO 表、手动块刷新逻辑对齐并抽象）。  
- **工程与版本**：工程文件结构、导入导出、与应用程序版本的兼容性策略。  
- **权限与安全**：谁能改页面、谁能下发写值、审计日志（可分阶段实现）。

### 2.2 非目标（除非另行立项）

- 完全复刻某一厂商组态软件的全部特性。  
- 在首版就实现 Web 组态与原生客户端 100% 特性对等（可作为后续阶段）。  
- 替代 PLC 编程软件；本平台聚焦 **HMI 与交互层**，PLC 代码生成可作为 **可选增值模块** 存在。

## 3. 核心概念模型

### 3.1 三层分离

1. **壳（Shell）**：主窗口、导航、主题、插件加载、全局服务（报警、日志、用户会话）。稳定、少变。
2. **页面与控件定义（Declarative UI）**：以序列化数据描述「有什么页面、页上有什么控件、如何布局、绑定到哪些数据点」。用户改动的主要是这一层。
3. **数据与规则（Data & Rules）**：OPC UA 节点、地址规范、IO 表、生成规则、模板。规则变更可触发 UI 或绑定关系的 **增量更新**。

### 3.2 关键对象

- **页面（Page）**：唯一 ID、标题、路由键、布局根节点引用、可选权限标记。  
- **控件实例（Widget Instance）**：类型 ID（对应控件库中的某一控件）、位置与尺寸（或网格坐标）、属性字典、**绑定说明**（如 OPC UA NodeId、方向、缩放）。  
- **控件库（Widget Library）**：每个控件提供默认属性 schema、设计时元数据、运行时 View/ViewModel 映射。  
- **工程（Project）**：页面集合 + 数据源配置 + 规则配置 + 资源（图片、样式覆盖）。  
- **生成单元（Generated Unit）**：由规则产生的一组页面片段或控件树（例如「某工位所有气缸」），带 **生成来源指纹**，便于重新生成时合并或提示冲突。

## 4. 功能性需求

### 4.1 页面管理

- 新增/复制/删除页面；支持排序与分组（如「手动」「自动」「参数」）。  
- 删除页面前 **依赖检查**（导航、脚本跳转、报警联动是否引用）。  
- 可选：页面模板（空白页、双列、带标题栏）。

### 4.2 控件增删与编辑（无源码）

- 从控件库 **拖拽或选取** 添加控件；选中后编辑属性与绑定。  
- 删除控件、撤销/重做（建议分阶段：先支持删除与简单撤销）。  
- 布局：锚定、网格、或容器树（与 WPF 能力匹配，避免一次承诺过度复杂的自由画布）。

### 4.3 数据绑定与写入策略

- 绑定类型：订阅值、读写、仅写（按钮脉冲）等。  
- 写值前 **确认、权限、互锁**（可配置）。  
- 与 OPC UA 的映射：集中维护连接、订阅、节点解析；页面只引用 **逻辑数据点 ID**，便于换站部署。

### 4.4 规则生成（与现有能力对齐）

现有实现已具备：**IO 表 → 解析对象类型 → 生成 PLC 文本 → 刷新手动界面气缸块/轴块**。平台化后应抽象为：

1. **输入模型**：IO 表行、注释规范、工位号等（与 `IoTableRow`、`IoGenerationSettings` 同类信息）。
2. **规则配置**：关键字到信号角色映射、对象类型识别、命名规则。
3. **输出分支**：
  - **PLC 程序产物**（保持现有生成物路径与模板机制）；  
  - **HMI 产物**：页面片段或控件实例列表 + 绑定到约定地址/节点。
4. **幂等与增量**：同一 IO 表再次导入时，能识别新增/删除/重命名，并更新 UI 或给出合并界面。

### 4.5 导入导出与迁移

- 工程导出为单文件或目录（含校验和）。  
- 版本号写入工程文件，壳程序加载时做 **兼容性检查** 与自动迁移（仅允许无损迁移）。

## 5. 非功能性需求

- **性能**：大量订阅时分组、节流；页面可见性变化时暂停不可见页订阅（可配置）。  
- **稳定性**：任一页控件异常不拖垮整个壳进程（控件隔离边界需在实现时权衡）。  
- **安全**：角色与权限、写操作审计、工程文件防篡改（可选签名）。  
- **可测试**：规则引擎与模板生成具备单元测试；核心 UI 模型可序列化反序列化回归测试。

## 6. 推荐技术架构（基于当前 WPF 栈）

### 6.1 UI 运行时

- **配置驱动**：主内容区根据当前页定义动态构造可视化树（`ContentControl` + `DataTemplateSelector` 或工厂模式）。  
- **控件注册表**：程序集内嵌或插件加载时注册 `WidgetId → 视图类型 + 默认 VM`。  
- **设计时与运行时**：首版可采用「独立组态编辑器 EXE」或「壳内设计模式」二选一；长期可并存。

### 6.2 数据层

- 延续 **OPC UA** 作为首选现场总线接入；抽象 `IDataSource` 以便后续增加 Simulation、MQTT 等。  
- 绑定解析器：将页面中的逻辑地址解析为实际 `NodeId` 与访问模式。

### 6.3 规则与模板

- **模板**：文本模板（现有 PLC 生成）+ **UI 模板**（如控件默认包、布局片段）。  
- **规则**：可数据化存储（JSON/YAML），关键逻辑可用小型 DSL 或 C# 脚本沙箱（需评估安全与许可证）。

### 6.4 存储

- 开发/单机：本地 JSON/SQLite。  
- 产线多客户端：可演进为集中存储 + 下发（后续阶段）。

## 7. 与当前代码库的映射（落地参考）


| 能力            | 现状                                                  | 平台化方向                         |
| ------------- | --------------------------------------------------- | ----------------------------- |
| 通信            | `OpcUaService` 等                                    | 归入数据适配层，页面只绑逻辑点               |
| IO 导入与 PLC 生成 | `IoTableImportService`、`IoProgramGenerationService` | 规则引擎公共内核；输出扩展为 **HMI 模型增量**   |
| 手动块 UI        | `RebuildManualCylinderBlocksFromIo` 等               | 改为对 **生成单元** 的合并更新，而非硬编码仅一种布局 |
| 配置持久化         | `ConfigurationService`、`AppConfig`                  | 拆分为 **应用设置** 与 **用户工程** 两类存储  |

## 8. 实施路线图（建议）

### 阶段 A — UI 模型与动态加载（基础）

- 定义页面/控件的 JSON schema；实现运行时工厂与最小控件集（文本、按钮、位图、数值绑定）。  
- 主窗口从静态 XAML 迁移为「导航 + 动态页宿主」。

### 阶段 B — 组态能力

- 提供页面与控件的增删改持久化；资源与主题覆盖。  
- 权限模型初版。

### 阶段 C — 规则生成一体化

- 将 IO 导入 → 对象识别 → PLC + HMI 联合生成纳入同一流水线；支持 diff 与冲突解决。  
- 扩展控件库（气缸、轴、真空等功能块组件化）。

### 阶段 D — 生态与工具

- 插件 API、第三方控件注册；工程云端备份与多用户协作（可选）。

---

## 9. 风险与决策点

- **自由组态 vs 实现成本**：全自由画布与脚本能力会显著增加测试与安全成本，建议按阶段放开。  
- **生成 UI 与手写 UI 混排**：需明确 **生成块** 的边界（只读区域 vs 可编辑区域），避免重复生成覆盖用户改动。  
- **许可证与代码保护**：若分发闭源壳 + 开放工程格式，需提前规划授权方案。
