## 阶段 V2-FIX · 收尾上一轮的尾巴（1–2 天，必须先做）

### V2-FIX-01 ⬜ 修补 V1 的 `catch { }`
- 范围：`ViewModels/MainViewModel.Manual.cs:634`
- 现状：`_ = Task.Run(async () => { ... } catch { })`，吞掉所有异常
- 改法：`catch (Exception ex) { Log.Warning(ex, "气缸状态二次刷新失败 block={Block}", block?.Name); }`
- 后续审计：grep `catch\s*\{\s*\}` 项目内为 0
- 验收：构建 0 警告；测试通过

### V2-FIX-02 ⬜ 修补 V1 的 `async void`
- **02a** `ViewModels/MainViewModel.Refresh.cs:352 UpdateAutoRefreshState` 不是事件处理器，改为 `async Task UpdateAutoRefreshStateAsync()`，所有调用点（4 处：MainViewModel.Monitor.cs:58、MainViewModel.cs:496/976/977/994）按上下文 await 或 fire-and-forget 包装。`partial void OnXxxChanged` 必须 fire-and-forget。
- **02b** `ViewModels/MainViewModel.Core.cs:102/107` Timer_Tick 和 `MainViewModel.Refresh.cs:437 OpcUaService_TagValueChanged` 是合法的事件处理器 async void，但**必须有顶层 try/catch + Log.Error**，否则未捕获异常会导致进程崩溃。
- 验收：grep `async void` 仅命中事件处理器；每个 async void 体内有 try/catch

## 阶段 V2-LINT · 清理 V1 遗留的 StyleCop 警告债务（3-5 天）

> 冷构建（`dotnet build` 强制全量）报 **3448 个 StyleCop 警告**，全是格式问题（V1-P3-CI-01 引入 StyleCop 但未清理）。增量构建被缓存掩盖，给人「0 警告」假象。

### V2-LINT-01 ⬜ 清理 StyleCop 警告
- 第一步：分类统计 `dotnet build /p:TreatWarningsAsErrors=false 2>&1 | grep -oE "SA[0-9]+" | sort | uniq -c | sort -rn`
- 第二步：高频规则评估 —— 决定每条规则是 **修代码** 还是 **降级到 None / Suggestion**（写入 `stylecop.json` 或 `.editorconfig`）
- 常见高频规则建议：
  - `SA1503`（强制大括号）→ 修代码
  - `SA1513 / SA1515 / SA1518`（空行/换行）→ 修代码或降级
  - `SA1116 / SA1137`（参数换行/缩进）→ 降级（噪声大）
  - `SA1101`（this 前缀）/ `SA1633`（文件头版权注释）→ 降级到 None
- 第三步：CI 上启用 `TreatWarningsAsErrors`，防止再次堆积
- 验收：`dotnet build` 警告 ≤ 50；CI 失败时能阻断 PR
