# 开放式 HMI 平台 — 实施步骤与任务分解

> **用途**：把《开放平台开发需求与架构》拆成可排期、可验收的开发步骤。  
> **执行方式**：按 **阶段 → 步骤编号** 顺序推进；同一步骤内可并行，**跨步骤** 注意依赖。  
> **关联文档**：`docs/开放平台开发需求与架构.md`

---

## 约定

- **DoD（完成定义）**：列出可演示或可自动化验证的结果。  
- **工程拆分**：`应用设置`（连接、窗口大小等）与 `用户工程`（页面、控件、绑定）尽早分文件，避免后期大迁移。  
- **主干策略**：每个阶段结束保留可运行、可回退的提交（或标签）。

## 阶段 0：开工准备（0.5～1 天）


| #   | 任务                                                                                             | 产出                    | DoD                |
| --- | ---------------------------------------------------------------------------------------------- | --------------------- | ------------------ |
| 0.1 | 冻结「阶段 A」范围：首版动态页只支持 **有限控件集**（文本、按钮、指示灯占位、数值只读）                                                | 一页纸范围说明（可写在本文档顶部变更记录） | 团队对「不做画布级自由拖拽」达成一致 |
| 0.2 | 仓库分支策略：如 `main` + `feature/runtime-ui`                                                         | 分支说明                  | 主分支始终可编译运行         |
| 0.3 | 定义工程文件目录约定：`config/app-settings.json` + `projects/<name>/project.json`（路径可调整，但 **应用/工程** 必须分离） | 目录约定写入本文档「附录」或 ADR    | 新建空工程目录可被程序识别      |


**依赖**：无。

## 阶段 A：UI 元模型 + 动态页宿主（核心基础）

目标：**用 JSON（或等价序列化）描述页面与控件，运行时渲染**；主窗口具备「导航 + 内容区」壳结构。

### A1 — 领域模型与序列化（1～2 天）


| #    | 任务                                                                                            | 产出                      | DoD                               |
| ---- | --------------------------------------------------------------------------------------------- | ----------------------- | --------------------------------- |
| A1.1 | 定义不可变核心类型：`ProjectDocument`、`PageDefinition`、`WidgetInstance`、`BindingSpec`（逻辑点 ID、访问模式、数据类型） | `Models/RuntimeUi/*.cs` | 可 `System.Text.Json` 序列化/反序列化单测通过 |
| A1.2 | 定义 `project.json` schema 版本字段 `schemaVersion`，并实现 **读时迁移钩子**（首版可为空实现）                         | 同上 + 最小迁移类              | 旧版本文件读入不抛未处理异常（或明确提示升级）           |
| A1.3 | 从现有 `AppConfig` / `ConfigurationService` 中 **识别** 哪些字段属于「应用设置」、哪些未来迁到工程（先文档化列表，不必一次搬完）        | 注释或 ADR                 | 清单经代码评审确认                         |


**依赖**：0.3。

### A2 — 控件注册表与运行时工厂（2～4 天）


| #    | 任务                                                                                         | 产出                                       | DoD                          |
| ---- | ------------------------------------------------------------------------------------------ | ---------------------------------------- | ---------------------------- |
| A2.1 | `IWidgetViewFactory` / `WidgetRegistry`：`widgetTypeId` → `DataTemplate` 或 `UserControl` 工厂 | `Services/RuntimeUi/` 或 `Views/Runtime/` | 单元测试：注册两个假控件，工厂能创建实例         |
| A2.2 | 实现首批控件：`text`、`bool-lamp`（占位）、`numeric-readonly`（占位）                                       | 对应 `UserControl` + VM                    | 从 `WidgetInstance` 绑定只读属性可显示 |
| A2.3 | 页面宿主控件：`DynamicPageHost` 递归根据子节点生成树（先 **纵向 Stack** 即可）                                     | XAML + code-behind 最小逻辑                  | 加载示例 `project.json` 能显示多控件   |


**依赖**：A1.1。

### A3 — 壳与导航接线（2～3 天）


| #    | 任务                                                                                                 | 产出        | DoD            |
| ---- | -------------------------------------------------------------------------------------------------- | --------- | -------------- |
| A3.1 | `MainWindow`（或新 `ShellWindow`）内容区改为：`Navigation` + `DynamicPageHost`；保留现有静态页为 **fallback**（开关或配置项） | XAML 调整   | 切换导航项可换页       |
| A3.2 | `RuntimeProjectService`（或等价）：启动时加载默认工程路径；暴露 `CurrentPage`、`Pages`                                  | VM + 服务注册 | 断点/日志可见加载成功    |
| A3.3 | 提供 **示例工程** `projects/_sample/project.json` 入库，开发环境默认指向它                                           | 示例文件      | 新同事克隆仓库一键看到动态页 |


**依赖**：A2.3。

### A4 — 逻辑数据点与 OPC UA 解析（3～5 天，可与 A3 部分并行）


| #    | 任务                                                                                   | 产出                      | DoD                               |
| ---- | ------------------------------------------------------------------------------------ | ----------------------- | --------------------------------- |
| A4.1 | `IDataPointCatalog`：逻辑 ID → `NodeId`、类型、读写；来源先为 **配置文件**（与 `appsettings`/专用 json 合并） | `Services/DataBinding/` | 修改 JSON 中 NodeId 后 UI 读值变化（真机或仿真） |
| A4.2 | 将现有 `OpcUaService` 订阅接口 **封装** 为按逻辑 ID 订阅（内部再映射 NodeId），避免页面直接依赖 UA 细节               | 适配层                     | 动态页控件订阅同一逻辑点不重复建订阅（或明确策略）         |
| A4.3 | 写操作：按钮控件 + `write-pulse` 或 `write-bool`（二选一先做最小）                                     | 控件 + 绑定枚举扩展             | 演示对仿真/测试节点写值成功                    |


**依赖**：A2.2、现有 `OpcUaService`。

**阶段 A 里程碑 DoD**：不修改 XAML 业务页源码，仅改 `project.json` 与数据点表即可增删控件并显示真实数据（读为主，写至少一种）。

## 阶段 B：组态持久化 + 页面生命周期（用户可改 UI）

目标：**在应用内**（或独立小工具）完成页面/控件的增删改，保存到工程文件；支持删除页面与基础校验。

### B1 — 工程 CRUD 与编辑器 VM（3～5 天）


| #    | 任务                                                                                 | 产出          | DoD                                |
| ---- | ---------------------------------------------------------------------------------- | ----------- | ---------------------------------- |
| B1.1 | `ProjectEditorService`：`AddPage` / `RemovePage` / `DuplicatePage` / `ReorderPages` | 服务 + 单测     | JSON 落盘后重开一致                       |
| B1.2 | `WidgetEditorService`：在指定页 `AddWidget` / `RemoveWidget` / `UpdateProperties`       | 同上          | 同页多控件 id 唯一性校验                     |
| B1.3 | 简易 **设计时面板**（可先列表 + 属性网格，不必拖拽）：选中页 → 控件列表 → 编辑属性                                   | 新 View 或对话框 | 操作后立即刷新预览区（`DynamicPageHost` 重载模型） |
| B1.4 | 删除页前 **引用检查**：导航目标、脚本占位（若尚无脚本，仅检查导航配置表）                                            | 校验器         | 被引用时禁止删除并提示                        |


**依赖**：阶段 A 里程碑。

### B2 — 资源与主题（可选压缩，2～3 天）


| #    | 任务                                      | 产出   | DoD                  |
| ---- | --------------------------------------- | ---- | -------------------- |
| B2.1 | 工程内 `assets/` 图片相对路径解析；运行时复制或 URI 加载    | 加载器  | 动态页 Image 控件可显示工程内图片 |
| B2.2 | 主题覆盖：工程级颜色/字体覆盖应用默认值（最小实现：几类 brush key） | 资源合并 | 切换工程可见不同主题           |


**依赖**：B1.1。

### B3 — 权限初版（可与 B1 并行，2～4 天）


| #    | 任务                                          | 产出    | DoD            |
| ---- | ------------------------------------------- | ----- | -------------- |
| B3.1 | 角色枚举：`Operator` / `Engineer` / `Admin`（可增减） | 模型    | 配置可持久化         |
| B3.2 | 页面级 `RequiredRole`；编辑模式仅 Engineer+          | VM 门禁 | 低权限看不到「编辑工程」入口 |
| B3.3 | 写值前确认对话框 + 角色要求（可配置关闭）                      | UI    | 演示账号切换行为差异     |


**依赖**：A4.2。

**阶段 B 里程碑 DoD**：非开发人员用内置编辑器完成「新增一页、加三个控件、保存、重启仍在」+「删除空页成功」。

## 阶段 C：规则生成与现有 IO 流水线合并

目标：IO 导入 / 对象识别后，除 PLC 文本外，**同步生成或更新 HMI 工程片段**；支持重复导入时的 **指纹与冲突提示**。

### C1 — 生成单元模型（2～3 天）


| #    | 任务                                                                                | 产出                | DoD          |
| ---- | --------------------------------------------------------------------------------- | ----------------- | ------------ |
| C1.1 | `GeneratedUnit`：`SourceFingerprint`（如 IO 表哈希+规则版本）、`TargetPageId`、`WidgetSubtree` | 模型                | 可序列化进工程      |
| C1.2 | 工程文件结构：手写控件与生成控件分区域存储（如 `generatedUnits[]`）                                       | `project.json` 扩展 | 加载后合并为单一可视化树 |
| C1.3 | 冲突策略枚举：`OverwriteGeneratedOnly` / `Prompt`（首版实现一种）                                | 合并服务              | 单元测试覆盖合并逻辑   |


**依赖**：阶段 B 至少完成 B1。

### C2 — 从 IO 规则到 HMI 产物（4～7 天）


| #    | 任务                                                                                                                                                    | 产出                        | DoD                       |
| ---- | ----------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------- | ------------------------- |
| C2.1 | 抽象 `IHmiGenerationSink`：输入 `IoTableRow` 列表 + 规则，输出 `GeneratedUnit` 列表                                                                                 | 接口 + 假实现                  | 与 UI 解耦可测                 |
| C2.2 | 把 `RebuildManualCylinderBlocksFromIo` / `RebuildManualAxisBlocksFromIo` 的逻辑 **迁到** 生成器：输出标准 `WidgetInstance` 树，而非直接操作 ObservableCollection（可保留旧路径作适配） | `Services/HmiGeneration/` | 生成结果与旧 UI 行为一致（对比截图或快照测试） |
| C2.3 | `IoProgramGenerationService` 或上层协调器：一次导入触发 **PLC 生成 + HMI 生成**（开关控制）                                                                                  | 编排                        | 日志可追踪两路结果                 |
| C2.4 | 重复导入：指纹相同跳过；指纹变化仅更新 `generated` 区域                                                                                                                    | 编排                        | 手工改的控件不被覆盖（策略与文档一致）       |


**依赖**：C1、`IoProgramGenerationService`、现有 IO 表模型。

### C3 — 规则外置与版本（2～3 天）


| #    | 任务                                          | 产出                    | DoD                     |
| ---- | ------------------------------------------- | --------------------- | ----------------------- |
| C3.1 | 规则配置 JSON：关键字→对象类型、模板版本号                    | `config/rules/*.json` | 修改规则无需改代码即可调整分类（在约定范围内） |
| C3.2 | 规则版本写入 `GeneratedUnit`，与 `schemaVersion` 协同 | 同上                    | 升级后可提示重新生成              |


**依赖**：C2.3。

**阶段 C 里程碑 DoD**：同一 IO 表导入两次，HMI 无重复控件；改 IO 后重新导入，生成块更新且手工块保留（按选定策略）。

## 测试与质量闸门（每阶段）

- **单元测试**：序列化、合并策略、WidgetRegistry、规则输出结构。  
- **冒烟**：连接 OPC UA 仿真 → 加载示例工程 → 切换页 → 写值。  
- **回归**：IO 导入后 PLC 生成物与当前版本 **diff** 在允许范围内（可加黄金文件测试）。
