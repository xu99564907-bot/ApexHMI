## 0. 总览

### 0.1 愿景

**让客户的 HMI 界面成为一份 JSON 文件，而不是 C# 源码。**

修改界面不需要重新编译 ApexHMI.exe，不需要 Visual Studio，甚至不需要程序员 —— 改 `projects/<客户>/project.json`，重启或点"重载工程"即可生效。

### 0.2 业务背景

ApexHMI 为不同客户的产线提供 HMI 界面。每个客户的设备配置不同：
- 客户 A：3 气缸 + 2 伺服 + 产量看板
- 客户 B：1 机械手 + 6 传感器 + 报警面板
- 客户 C：纯监控页面

**V1 现状（已有设计器）**：V1 已实现 JSON 驱动的设计器系统（`DesignerProject` → `DesignerPage` → `DesignerElement`），支持 11 种工业控件（Button / Indicator / Label / ValueDisplay / AlarmBanner / Motor / Cylinder / Axis / Robot / Stopper / PageButton），具备拖拽设计、设计/运行双模式、实时 OPC UA 值更新、页面跳转等能力。工程保存为 `designer-project.json`，无需重编译即可生效。

**V1 的局限**：
- 控件渲染集中在 `MainViewModel.UpdateRuntimeVisuals()` 的 switch-case 中，缺乏工厂模式和独立 ViewModel，扩展新控件需改动主 ViewModel
- Tag 绑定直接使用字符串 TagName + CommandBinding，没有 BindingSpec 这样的结构化绑定模型
- TagId（逻辑名）与 NodeId（物理地址）未解耦，换 PLC 地址需改 tags.json 中所有引用
- 无 Schema 版本管理，工程格式变更时缺乏升级路径

**开放平台（Phase A）做法**：在 V1 基础上进行架构重构——用 `WidgetRegistry` 工厂模式替代 switch-case，用 `BindingSpec` 结构化绑定模型替代字符串拼凑，用 `DataPointCatalog` 实现 TagId↔NodeId 解耦，用 `RuntimeDataBindingService` 独立管理 OPC UA 订阅生命周期，用 `ProjectMigration` 实现工程文件版本升级。所有客户共用**同一个** ApexHMI.exe，每个客户拿到独立的 `project.json`（格式与 V1 的 `designer-project.json` 不同，需迁移）。

### 0.3 阶段划分

| 阶段 | 目标 | 状态 | 与 V1 关系 |
|---|---|---|---|
| Phase A — 运行时 | 用 WidgetRegistry / BindingSpec / DataPointCatalog 架构重构 V1 的 JSON 驱动渲染管线 | ✅ | 替代 V1 的 `UpdateRuntimeVisuals()` switch-case 模式 |
| Phase B — 编辑器 | 改造 V1 已有的设计器，接入 Phase A 的 WidgetRegistry + DataPointCatalog | ⬜ | 改造 V1 的 DesignerView / DesignerElement 属性面板 |
| Phase C — IO→HMI 一体化 | 从 IO 表自动生成对应控件页（气缸→气缸控件，伺服→伺服控件） | ⬜ | 消费现有 IoTableImportService |

### 0.4 术语表

| 术语 | 含义 |
|---|---|
| **工程（Project）** | 一份 `project.json` + 关联 assets，对应 `ProjectDocument` |
| **页面（Page）** | 一个 HMI 视图，对应 `PageDefinition`（含 RouteKey、画布大小、控件列表） |
| **控件（Widget）** | 页面上的一个 UI 元素，对应 `WidgetInstance`（类型 + 位置 + 属性 + 绑定 + 动作） |
| **TypeId** | 控件类型标识符（"text" / "bool-lamp" / "button" 等），由 `WidgetRegistry` 注册 |
| **TagId** | 逻辑变量名，对应 `tags.json` 中 `TagItem.Name`，与具体 OPC UA NodeId 解耦 |
| **RouteKey** | 页面路由键（"overview" / "manual"），按钮通过它跳转目标页 |

### 0.5 V1 与 Phase A 模型对照

| 概念 | V1 模型 | Phase A 模型 | 迁移方向 |
|---|---|---|---|
| 工程 | `DesignerProject` | `ProjectDocument` | V1→Phase A 需格式转换 |
| 页面 | `DesignerPage` | `PageDefinition` | 结构相似，字段名不同 |
| 控件/元素 | `DesignerElement`（强类型属性） | `WidgetInstance`（`Dictionary<string,string>` 属性） | 属性名映射 |
| 标签绑定 | `TagBinding: string` + `CommandBinding: string` | `BindingSpec{TagId, AccessMode, DataType}` | 字符串→结构化 |
| 跳转目标 | `NavigationTarget: string` | `ActionType="navigate"` + `ActionParam` | 语义等价 |
| 控件注册 | 硬编码 switch-case in `UpdateRuntimeVisuals()` | `WidgetRegistry.Register(typeId, factory)` | 工厂模式 |
| 持久化格式 | `designer-project.json` | `project.json` | 两份格式并存，需迁移器 |

## 1. Phase A — 运行时（已完成）

> **V1 对照**：Phase A 是对 V1 已有 JSON 驱动设计器的架构重构。V1 的 `UpdateRuntimeVisuals()` 方法（[MainViewModel.cs#L1261-L1303](../ViewModels/MainViewModel.cs#L1261-L1303)）已能根据 OPC UA 值实时更新 11 种控件的颜色和文字，但实现方式是集中式 switch-case。Phase A 将其拆解为 WidgetRegistry 工厂 + 独立 WidgetViewModel + RuntimeDataBindingService，获得更好的可扩展性和可测试性。

### 1.1 总体架构

```
┌─────────────────── ApexHMI.exe ───────────────────┐
│                                                   │
│  RuntimeProjectService                            │
│       │ 加载 project.json                         │
│       ▼                                           │
│  ProjectDocument ─► PageDefinition ─► WidgetInstance │
│                                          │        │
│  DynamicPageHostViewModel ◄──────────────┘        │
│       │ 调 WidgetRegistry.Create(typeId)          │
│       ▼                                           │
│  四种内置控件：text / bool-lamp / numeric / button│
│       ▲                                           │
│       │ 值推送                                    │
│  RuntimeDataBindingService                        │
│       │ 订阅                                      │
│       ▼                                           │
│  IOpcUaService ─► PLC                             │
│                                                   │
│  DataPointCatalog ─► tags.json（TagId → NodeId）  │
└───────────────────────────────────────────────────┘
```

### 1.2 数据模型（[Models/RuntimeUi/](../Models/RuntimeUi/)）

```csharp
ProjectDocument                     // project.json 根对象
├── SchemaVersion : int             // 当前 = 1，由 ProjectMigration 升级
├── ProjectName : string
├── DefaultPageRouteKey : string?   // 启动默认页
└── Pages : List<PageDefinition>
        ├── Id / Title / RouteKey
        ├── RequiredRole : string?  // null = 所有人可见
        ├── CanvasWidth / CanvasHeight  (绝对定位画布)
        └── Widgets : List<WidgetInstance>
                ├── Id / TypeId
                ├── X / Y / Width / Height
                ├── Properties : Dictionary<string,string>
                ├── Binding : BindingSpec?
                │       ├── TagId           // 引用 TagItem.Name，不写死 NodeId
                │       ├── AccessMode      // Subscribe / ReadOnly / ReadWrite / WriteOnly
                │       └── DataType        // Bool / Int / Float / String
                ├── ActionType : string?    // "write-bool" / "write-pulse" / "navigate"
                └── ActionParam : string?   // 写入值 或 跳转目标 RouteKey
```

**设计要点**：
- `Properties` 是字符串字典，每种控件自行约定 key —— 牺牲强类型换可演化性（加新属性不破坏 JSON 兼容性）
- `Binding.TagId` 引用**逻辑名**而非 NodeId，因此换 PLC / 换地址不影响工程文件
- `SchemaVersion` 配合 `ProjectMigration.Migrate()` 支持读时升级，老工程文件可被新版本无损打开

### 1.3 服务层

#### RuntimeProjectService（[`Services/RuntimeProjectService.cs`](.claude/worktrees/exciting-montalcini-582697/Services/RuntimeProjectService.cs)）
- `LoadDefault()` — 加载 `<exe>/projects/_sample/project.json`，不存在时自动生成两页演示工程
- `Load(path)` — 反序列化 + 调用 `ProjectMigration.Migrate`
- `Save(doc, path)` — 序列化为缩进 JSON，UTF-8 编码

#### WidgetRegistry（[`Services/RuntimeUi/WidgetRegistry.cs`](.claude/worktrees/exciting-montalcini-582697/Services/RuntimeUi/WidgetRegistry.cs)）
控件类型 → 工厂委托的注册表：
```csharp
Register("text",             (m, ctx) => CreateView(new TextWidgetViewModel(m, ctx),             new TextWidget()));
Register("bool-lamp",        (m, ctx) => CreateView(new BoolLampWidgetViewModel(m, ctx),        new BoolLampWidget()));
Register("numeric-readonly", (m, ctx) => CreateView(new NumericReadonlyWidgetViewModel(m, ctx), new NumericReadonlyWidget()));
Register("button",           (m, ctx) => CreateView(new ButtonWidgetViewModel(m, ctx),          new ButtonWidget()));
```
未知 TypeId 返回灰色占位 `[未知控件]`，不会让整页崩溃。

#### DataPointCatalog（[`Services/DataBinding/DataPointCatalog.cs`](.claude/worktrees/exciting-montalcini-582697/Services/DataBinding/DataPointCatalog.cs)）
- 启动时从 `config/tags.json` 加载 `TagItem` 列表
- `Merge(IEnumerable<TagItem>)` 接受外部刷新（如 IO 表导入后）
- `FindTag(tagId)` 查表

#### RuntimeDataBindingService（[`Services/DataBinding/RuntimeDataBindingService.cs`](.claude/worktrees/exciting-montalcini-582697/Services/DataBinding/RuntimeDataBindingService.cs)）
- `AttachAsync(host)` — 切换页面时为页面中的 TagId 建立 OPC UA 订阅，已订阅过的不重复
- `WriteAsync(tagId, value)` — 主动写值
- 监听 `IOpcUaService.TagValueChanged` 事件，回调 `host.PushTagValue` 推到对应 widget

### 1.4 ViewModel 层（[`ViewModels/Runtime/`](.claude/worktrees/exciting-montalcini-582697/ViewModels/Runtime/)）

**WidgetViewModelBase** — 各 Widget VM 基类，提供 `RegisterValueCallback` 抽象访问。

四个具体 VM：
- `TextWidgetViewModel` — 静态文本（不订阅）
- `BoolLampWidgetViewModel` — Bool 指示灯，按 trueColor/falseColor 切换
- `NumericReadonlyWidgetViewModel` — 数显（含 unit / format）
- `ButtonWidgetViewModel` — 按钮，点击触发 `IWidgetDataContext.ExecuteAction(ActionType, ActionParam)`

**DynamicPageHostViewModel** — 页面宿主：
- 持有当前 `PageDefinition`
- 调 `IWidgetViewFactory.Create()` 实例化每个控件视图，按 `X/Y` 包成 `PositionedWidget`
- 实现 `IWidgetDataContext`：维护 `tagId → List<Action<string>>` 回调表，`PushTagValue(tagId, value)` 时分发
- `BoundTagIds` 暴露当前页所有需订阅的 TagId（供 `RuntimeDataBindingService` 使用）

### 1.5 视图层（[`Views/Runtime/`](.claude/worktrees/exciting-montalcini-582697/Views/Runtime/)）

- `DynamicPageHost.xaml` — `Canvas` 宿主，`ItemsControl` 绑定 `WidgetElements`，每项按 `X/Y` 设置 `Canvas.Left/Top`
- `Widgets/TextWidget.xaml` / `BoolLampWidget.xaml` / `NumericReadonlyWidget.xaml` / `ButtonWidget.xaml` — 四个独立 UserControl

页面入口 [`Views/Pages/RuntimeDesignerView.xaml`](.claude/worktrees/exciting-montalcini-582697/Views/Pages/RuntimeDesignerView.xaml) 改造为：
- 顶部：页面下拉选择 + "重载工程"按钮
- 中部：嵌入 `DynamicPageHost`

### 1.9 已知限制（Phase A 未做、留给 B/C 或后续阶段）

- ❌ **控件库小**：仅 4 种基础控件，V1 已有 11 种（Motor / Cylinder / Axis / Robot / Stopper / AlarmBanner / Indicator / Label / ValueDisplay / Button / PageButton），Phase A 尚未覆盖 Motor / Robot / Stopper / Indicator / Label / ValueDisplay / PageButton（→ Phase B-10 需补全，目标覆盖 V1 全部 11 种）
- ❌ **无逻辑编排**：按钮只能写一个值或跳一页，做不了"按下后写一组值并等 1s 再读回"（→ 后续阶段，可能 Phase D）
- ❌ **assets 不解析**：图片/图标不能由工程引用（→ Phase B 顺带）
- ❌ **主题不可覆盖**：颜色硬编码在 properties，不能定义工程级主题（→ Phase B 顺带）
- ❌ **角色拦截未实现**：`RequiredRole` 字段存在但运行时未校验（→ Phase B-04）
- ❌ **无单元测试**：33 个文件 0 测试（→ V2-MERGE 合并后必须补）
- ❌ **未接入 V1 治理基础设施**：`DataPointCatalog.TryLoadFromConfigFile` 用空 catch、未用 Serilog；不用 IOptions（→ 合并时随手修）
- ❌ **V1 工程格式未迁移**：V1 用户已有 `designer-project.json`，尚无 V1→Phase A 格式转换工具（→ 见 1.10）

## 2. Phase B — 编辑器（待开发）

> **V1 对照**：V1 已有完整的图形化设计器（`DesignerView.xaml` + 拖拽 Canvas + 属性面板），Phase B 不是新建编辑器，而是将 V1 设计器**接入 Phase A 架构**——用 WidgetRegistry 替代 `UpdateRuntimeVisuals()` 的 switch-case，用 DataPointCatalog + BindingSpec 替代字符串拼凑的 TagBinding/CommandBinding，统一为 Phase A 的 `DynamicPageHost` 渲染。

### 2.1 目标

- 属性面板编辑 `WidgetInstance.Properties` 字典而非 `DesignerElement` 强类型属性

### 2.3 关键设计决策

#### B-D1 编辑模型 vs 运行模型
**决策**：保持单一模型 `ProjectDocument`。编辑器只是读写它的服务，不引入"编辑器专用模型"。
**理由**：避免双模型同步成本；改完即时预览。

#### B-D2 数据流方向
```
[属性网格] ──UpdateProperty──► [WidgetEditorService] ──修改─► [WidgetInstance]
                                                                  │
                                                                  ▼
                              [DynamicPageHostViewModel] ◄──── PropertyChanged
                                       │
                                       ▼
                                  [Widget VM 重读]
```
**决策**：`WidgetInstance` 升级为 `ObservableObject`（CommunityToolkit.Mvvm）。
**理由**：让属性面板改完即时反映到画布。

#### B-D3 保存策略
**决策**：自动保存（每次编辑 1.5s debounce 后写盘），并保留显式"保存"按钮。
**理由**：现场不易记得手动保存；但保留按钮供仪式感。

#### B-D4 设计 / 运行模式开关
**决策**：复用 V1 已有的 `MainViewModel.IsRuntimeMode`，运行 / 设计两个模式只切显示和命令可用性，不切页。
**理由**：避免双窗口；改完立即在同一 Canvas 上点按钮验证。

## 3. Phase C — IO→HMI 一体化（待开发）

> **V1 对照**：V1 已实现 IO 表导入 → PLC 程序生成（`IoProgramGenerationService`），但没有自动生成 HMI 页面。Phase C 填补这个空缺，且直接生成 Phase A 格式的 `WidgetInstance`，享受 WidgetRegistry 工厂模式和 DataPointCatalog 解耦的好处。

### 3.1 目标

把现有 [IoTableImportService](../Services/IoTableImportService.cs) 导入的 IO 表（含气缸 / 伺服 / 传感器 / 输出），**一键生成**对应的开放平台工程页面。

#### 现状问题
- IO 表导入后只生成了 PLC 程序（IoProgramGenerationService），**没有自动生成 HMI 页面**
- 工程师还要手写 JSON：每个气缸要 3-5 个 widget（指示灯 + 前进按钮 + 退回按钮 + 互锁灯）
- 出错率高、效率低

### 3.2 工作流目标

```
                  IoTable.csv
                       │
                       ▼
            IoTableImportService（已有）
                       │
                       ▼
              IoTableParseResult（已有）
                       │
                       ▼
          ┌────────────┴────────────┐
          ▼                         ▼
  IoProgramGenerationService   HmiGenerationService（Phase C 新增）
  （PLC .st 文件，已有）         │
                                ▼
                         ProjectDocument 增量
                                │
                                ▼
                       IProjectEditorService.MergeFrom(...)
```

### 3.3 任务分解

| ID | 任务 | 输入 | 输出 | 验收 |
|---|---|---|---|---|
| **C-01** | `Services/Generation/IHmiGenerationRule` | IoTableRow / 块类型 | 生成一个 `IEnumerable<WidgetInstance>` + 可选页 | 每条 rule 单元测试 |
| **C-02** | 内置规则：气缸块 → 6 个控件 | ManualCylinderBlockItem | 标题 / 到位灯 / 前进按钮 / 退回按钮 / 互锁灯 / 错误灯 | 演示 IO 表导入后气缸页正确生成 |
| **C-03** | 内置规则：伺服块 → ? 个控件 | ManualAxisBlockItem | 当前位置数显 / 速度数显 / 启动 / 停止 / 复位 | 同上 |
| **C-04** | 内置规则：纯 IO 点 → 灯 + 按钮 | IoTableRow（输入或输出） | bool-lamp 或 button | 同上 |
| **C-05** | 自动布局算法 | 控件列表 | X/Y 自动排版（grid / pack / 流式），避免重叠 | 1000 个控件分页生成无重叠 |
| **C-06** | 命名归一化 | RuleSet | 沿用 NamingRulesService，使 TagId 与 IO 表一致 | TagId 解析正确 |
| **C-07** | `IProjectEditorService.MergeFrom(generatedDoc, MergeStrategy)` | 现有 + 生成 | 三种策略：覆盖 / 追加新增 / 仅补缺失 | 测试三种策略 |
| **C-08** | 一键命令 UI | DesignerViewModel | "从当前 IO 表生成 HMI 页面"按钮 | 端到端：导 IO 表 → 点按钮 → 工程文件已增加对应页 |
| **C-09** | 增量再生成保留手动改动 | 二次生成时，已被人工改过 X/Y/Properties 的控件不被覆盖 | 引入 `WidgetInstance.GeneratedFingerprint` | 测试：修改后再生成不丢失改动 |

### 3.4 关键设计决策

#### C-D1 生成 vs 模板
**决策**：用代码规则（IHmiGenerationRule 实现）而不是 XAML / mustache 模板。
**理由**：HMI widget 间的相对位置约束、命名规则等用代码表达更清晰；模板适合纯文本生成。

#### C-D2 一次性 vs 增量
**决策**：增量生成（C-09），通过 fingerprint 区分"机器写的"vs"人工改的"。
**理由**：现场实际场景就是改了 IO 表后追加生成几个控件，不应清掉之前的人工调整。

#### C-D3 与 PLC 程序生成的关系
**决策**：HmiGenerationService 与 IoProgramGenerationService **平行**，都消费 IoTableParseResult。
**理由**：解耦；某些客户可能只要 HMI 不要 PLC 代码（监控屏）。

### 3.5 验收标准（端到端）

1. 工程师导入一份含 5 个气缸的 IO 表
2. 点 "从 IO 表生成 HMI 页面"
3. project.json 中出现新页 "气缸控制（自动生成）"，含 30 个控件（5 × 6）正确排版
4. 切到运行模式，所有 widget 已绑定到正确的 OPC UA Tag，能写值能读回
5. 工程师手动调整其中一个气缸的按钮 X 坐标，保存
6. 再次导入更新过的 IO 表（多了 1 个气缸）+ 点生成
7. 已存在的 5 个气缸保留人工改动，新增第 6 个气缸自动生成

## 6. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| **双轨制（最严重）**：Phase A 与 V1 两套设计器模型并存 | 维护两份代码、两份 JSON 格式、用户困惑 | 1.10 迁移策略：V2-MERGE-01 阶段完成 V1→Phase A 自动迁移，统一为 Phase A 模型；V1 的 `DesignerElement` 逐步废弃 |
| Phase A 没有单元测试 | 后续重构难发现回归 | V2-MERGE-01 合入时强制补测试 |
| `Properties` 字典是字符串 → 字符串 | 类型错误运行时才暴露 | B-06 属性网格做类型校验，警告非法值 |
| 工程文件膨胀（数千控件） | 加载慢、JSON 难审 | C-05 分页 + 工程文件按页拆分（Phase B+） |
| OPC UA Tag 改名后所有工程 broken | 客户工程失效 | 提供 "Tag 重定向"工具（B-07 顺带） |
| 角色拦截绕过（直接改 JSON） | 安全风险 | B-04 在加载层而非 UI 层校验 |
| 自动生成覆盖人工改动 | 现场改动丢失 | C-09 fingerprint 隔离 |
| **V1 控件功能倒退**：Phase A 控件不如 V1 丰富 | Phase A 无法独立使用 | B-10 一次性覆盖全部 11 种 V1 控件类型，不遗漏 |
